# Recover24

ボイスフィッシング・金融詐欺被害者の自然言語による申告内容を構造化し、銀行担当者が確認できる文書パッケージと内部レビューキューを生成する、LLMベースの被害受付・文書化プロトタイプです。

本プロジェクトは、「被害者がチャットボットに話せば、そのまま補償される」という自動判断システムではありません。被害者の断片的な事実関係を整理し、銀行担当者が迅速に確認できるよう、**事件の構造化、不足情報に対する質問、提出準備度の確認、文書生成、監査ログ**を分離した運用支援システムです。

---

## 1. Overview

ボイスフィッシング被害者は、被害直後に何を優先すべきか、銀行・警察・通信会社・カード会社へどの情報を提出すべきかを判断することが困難です。

被害者は多くの場合、緊張した状態で事件を説明しなければなりません。一方、相談担当者は、被害時刻、送金内容、なりすまし相手の情報、不正アプリのインストール有無、身分証・OTP・口座情報の漏えい有無、支払停止依頼の有無、証拠資料の保有状況を短時間で確認する必要があります。

Recover24はこの問題を、**被害者の自然言語による申告を、銀行で受付可能な事件構造へ変換する問題**として定義しました。

中心的な処理フローは次のとおりです。

```text
被害者の自然言語による申告
→ LLMによる初期事実抽出
→ RecoveryCaseV1事件JSONの生成
→ ルールエンジンによる不足フィールド／リスクシグナル／次の質問の計算
→ 明示的な質問IDに基づく追加回答の反映
→ 文書提出準備度の確認
→ 被害救済／警察届出／銀行相談用文書パッケージの生成
→ 銀行担当者向けレビューキューとコンソールViewModelの生成
→ Audit Trailの記録
```

Recover24の中心は、LLMが最終判断者にならないよう役割を分離することです。

LLMは、被害者の自由記述から事実候補を抽出するために使用します。一方、不足フィールド、次の質問、提出準備度、証拠チェックリスト、文書出力は、構造化されたcase modelとルールベースのロジックで管理します。

したがって、本プロジェクトの結論は次のとおりです。

> LLMは被害受付の判断者ではなく、被害者の非構造化された申告を、銀行担当者が確認可能な事件パッケージへ整理する補助レイヤーであるべきである。

## 2. Problem & Objective

ボイスフィッシング被害の受付は、単純なFAQチャットボットでは解決しにくい問題です。

被害者が「詐欺に遭った」と話しても、実際に銀行が確認すべき情報は事件類型によって大きく異なります。

例えば、家族になりすましたメッセンジャーフィッシングでは、身分証・カード・口座情報の漏えい有無や遠隔操作アプリのインストール有無が重要です。融資を装った詐欺では、現金の引き渡し、指定口座への送金、融資実行の有無、なりすました機関の情報が重要です。不正アプリや名義盗用が含まれる場合は、通信会社への届出、名義盗用による口座・携帯電話の開設有無、不正融資・カードローンの有無まで確認する必要があります。

Recover24が解決しようとする問題は次のとおりです。

| 課題 | 従来方式の難しさ | Recover24のアプローチ |
|---|---|---|
| 被害申告が非構造的 | 被害者が時系列・項目別に説明することが難しい | 自然言語による申告を事件JSONへ構造化 |
| 事件類型ごとに必要な質問が異なる | 固定質問票だけでは不足が生じる | 事件類型と不足フィールドに応じて次の質問を生成 |
| 提出書類が複雑 | 被害者が準備すべき資料を把握しにくい | 証拠チェックリストと文書準備度を計算 |
| 相談担当者の確認負担が大きい | 長い申告内容を毎回読み直し、整理する必要がある | Case Brief、相談メモ、提出文書パッケージを生成 |
| LLMの誤判断リスク | 誤った自動判断が被害者をさらに混乱させる可能性 | LLM抽出とルール／文書／人による確認を分離 |
| 個人情報・機微情報が多い | ログ・出力・保管ポリシーが重要 | 監査ログ、マスキング、人による確認を前提 |

本プロジェクトの目的は3つです。

第一に、被害者の自然言語による申告を`RecoveryCaseV1`構造へ変換します。

第二に、事件状態に基づいて不足情報と次の質問を生成し、被害受付を段階的に完成させます。

第三に、銀行担当者が確認できる文書パッケージ、レビューコンソール用データ、監査ログを生成します。

Recover24は、112への届出や銀行の正式な審査を代替しません。銀行のWeb／アプリへ搭載できる**事前受付・文書化レイヤー**として設計しました。

## 3. Data

Recover24では実際の顧客データを使用せず、ボイスフィッシング事例を基にしたゴールデンセットと合成補強シナリオを使用しました。

評価用ゴールドデータセットは、34件の被害事例確認表を基に構成しました。その中で、個別の被害受付シナリオとして評価可能な事例と統計的資料を区別し、すべての事件を人による確認が必要な受付事例として扱いました。

| 項目 | 内容 |
|---|---|
| 評価事例数 | 34件 |
| 主な事件類型 | メッセンジャーフィッシング、融資詐欺、機関なりすまし、スミッシング、不正アプリ、名義盗用、対面詐取 |
| 主な接触チャネル | SMS、カカオトーク、電話、インターネット広告、インターネットコミュニティ |
| 主な被害形態 | 口座振込、現金引き渡し、カードローン・キャッシング、名義盗用口座／携帯電話、不正アプリのインストール |
| 評価原則 | 原文に基づく事実とデモ用補強値を区別 |
| 運用判断 | すべてのケースでhuman reviewが必要 |

評価データで重視した情報は次のとおりです。

| 情報グループ | 例 |
|---|---|
| 事件類型 | 家族なりすまし、融資なりすまし、公的機関なりすまし、不正アプリ、名義盗用 |
| 接触チャネル | SMS、電話、カカオトーク、インターネット広告 |
| 被害金額 | 振込金額、現金引き渡し金額、カードローン／融資実行額 |
| 漏えい情報 | 身分証、口座情報、カード情報、暗証番号、OTP、証明書 |
| 不正アプリ／遠隔操作 | 不正アプリのインストール有無、TeamViewerなど遠隔操作アプリの有無 |
| 名義盗用リスク | 携帯電話開通、口座開設、不正融資、カードローン、少額決済 |
| 事後対応 | 支払停止依頼、警察届出、通信会社届出、相談番号 |
| 証拠資料 | SMS／カカオトークのスクリーンショット、通話履歴、振込確認書、身分証コピーなど |

本プロジェクトで重要なのは、「正解データに合わせて文章をきれいに作ること」ではなく、被害受付業務に必要な事実が欠落しないようにすることです。

そのため、評価は一般的なチャットボットの回答品質ではなく、必須事実の抽出、不足質問、禁止質問、文書準備度、人による確認の必要性を中心に設計しました。

## 4. Method / System Design

Recover24の中心的な設計原則は、**事件状態を一つに保ち、各レイヤーの責任を分離すること**です。

初期バージョンでは、追加回答を反映する際に事件全体を再生成する流れがありました。この構造では、日付質問を一つ修正しただけで証拠状態が消えたり、被害類型の修正が文書文言や評価文字列の失敗につながったりする可能性がありました。

そこで構造を次のように整理しました。

```text
RecoveryCaseV1
  → deep copy
  → 対象フィールドのみpatch
  → 派生値をrefresh
  → invariant検証
  → 新しいRecoveryCaseV1を返す
```

中心的なレイヤーは次のとおりです。

| レイヤー | 責任 | 禁止したこと |
|---|---|---|
| `models.py` | Pydantic事件モデルと型定義 | 質問生成、LLM呼び出し、文書出力 |
| `catalogs.py` | 被害類型・証拠・銀行名の正規化catalog | case変更、HTML生成 |
| `extraction.py` | 初回自然言語申告から構造化事実を抽出 | 追加回答処理、文書出力 |
| `intake/questions.py` | 不足フィールドに基づく質問生成 | ユーザー回答の解析、case変更 |
| `intake/answer_router.py` | 質問のhandler_idによる回答処理ルーティング | 質問文キーワードからhandlerを推測 |
| `intake/answer_handlers.py` | 日付・取引・証拠など対象フィールドのみpatch | case全体の再生成 |
| `case_rules.py` | 被害総額、不足フィールド、次の行動など派生値の再計算 | ユーザー入力値の初期化 |
| `documents/view_model.py` | 文書表示用ViewModelを生成 | case変更、被害類型の再分類 |
| `documents/html_renderer.py` | ViewModelに基づくHTML出力 | 事件推論、証拠状態の変更 |
| `evaluation/` | ゴールデンセットsimulationとscoring | 製品内部ロジックへの直接依存 |

この設計では、`RecoveryCaseV1`が唯一の事件状態です。`ExtractedIncidentV1`は初回LLM抽出結果を渡す一時DTOとしてのみ使用し、追加回答の処理では使用しません。

質問も文面そのものでルーティングしません。各質問には`question_id`、`field_path`、`handler_id`、`category`があります。

例えば日付質問は次のように明示的に定義されます。

```text
question_id: timeline.first_discovered_at
field_path: accidentInfo.firstDiscoveredAt
handler_id: timeline.first_discovered_at
category: timeline
```

この構造により、質問文が多少変わっても、評価器や回答処理器が質問種類を誤認しません。

## 5. Implementation

Recover24は単一のチャットボット関数ではなく、受付フローを複数モジュールに分けて実装しています。

実装フローは次のとおりです。

1. 被害者の初回自然言語申告を入力します。
2. LLM providerが構造化JSON候補を生成します。
3. `extraction.py`が抽出結果を検証し、`RecoveryCaseV1`を生成します。
4. `case_rules.py`が不足フィールド、リスク度、次の行動、証拠チェックリストを計算します。
5. `questions.py`が`QuestionSpec`に基づく次の質問を生成します。
6. 被害者が追加質問へ回答すると、`answer_router.py`がhandlerを選択します。
7. `answer_handlers.py`が該当フィールドのみをpatchします。
8. `refresh_case()`が派生値のみを再計算します。
9. `DocumentViewModel`を生成します。
10. HTML文書、Case Brief、相談メモ、reviewer console viewを生成します。
11. `AuditTrail`へ変更履歴を記録します。

主な実行フローは次のように理解できます。

```text
start_case()
  → analyze_initial_statement()
  → build_case_from_extraction()
  → refresh_case()
  → build_questions()
  → apply_answer()
  → build_document_view_model()
  → render_case_html()
  → write_case_artifacts()
```

文書出力はRecover24の重要な機能です。生成対象は単一書類ではなく、銀行・警察・被害救済申請に必要な情報をまとめた確認パッケージです。

| 出力文書／ビュー | 含まれる情報 |
|---|---|
| 警察届出用の被害内容整理文 | 事件概要、なりすまし相手情報、被害日時、振込内容、被害経緯、届出理由、証拠資料 |
| 被害救済申請書用の情報要約 | 被害者情報、振込内容、詐欺利用口座情報、被害救済依頼理由 |
| 電子金融取引事故被害届の補助資料 | 事故発生／認知／支払停止時刻、取引内容、事故経緯、事後対応 |
| 銀行／112支払停止電話メモ | 被害額、出金銀行、入金銀行／口座、送金時刻、支払停止依頼の有無 |
| Reviewer Console View | 担当者向け事件要約、不足情報、リスクシグナル、次の対応 |
| Audit Trail | 事件生成、フィールド変更、文書生成履歴 |

これらの文書は正式様式を代替するものではありません。実際の提出前には、金融会社、警察、関係機関の最新様式と案内を確認する必要があります。

## 6. Evaluation

Recover24の評価は、一般的なチャットボット回答品質ではなく、**被害受付業務の完成度**を基準に設計しました。

重要な評価指標は次のとおりです。

| 評価基準 | 確認内容 | 理由 |
|---|---|---|
| Critical Field Extraction | 被害金額、取引、漏えい情報、不正アプリ、名義盗用リスクなど中心事実の抽出 | 受付文書と対応判断の基盤 |
| Critical Missing Field Recall | 必ず追加質問すべき不足情報の捕捉 | 被害者へ必要な質問を漏らさないため |
| Next Question Quality | 事件類型に合った次の質問の生成 | 固定質問の繰り返しを減らすため |
| Forbidden Question Avoidance | 既に判明している質問や不適切な質問の反復を防止 | 被害者体験と受付効率の改善 |
| Document Completion Rate | 文書提出に必要なフィールドがどの程度入力されたか | 銀行担当者が確認可能かを判断 |
| Human Review Routing | 人による確認が必要か | 自動判断リスクの防止 |
| Hallucination / Unsupported Claim | 原文にない事実を生成していないか | 機微な事件における虚偽情報の防止 |
| Auditability | どの回答によりどのフィールドが変更されたかを追跡 | 運用確認と障害分析 |

ゴールデンセットの確認では、すべてのケースが人による確認を必要とする事件に分類されました。これはRecover24が、自動補償・自動判断システムではなく、銀行担当者へ確認可能な事件パッケージを渡す構造であることを示しています。

初期評価とリファクタリングの過程で確認した代表的な失敗モードは次のとおりです。

| 失敗モード | 問題 | 改善方針 |
|---|---|---|
| 振込型質問の反復 | 不正アプリ・名義盗用事件でも出金銀行／入金銀行／送金時刻だけを反復質問 | 事件類型別question groupと`QuestionSpec`を導入 |
| 追加回答反映時のcase再生成 | 日付一つの修正でも証拠状態や他フィールドが損傷する可能性 | handlerベースの対象フィールドpatchへ変更 |
| 質問文ベースの評価 | 質問文が変わると評価器が質問種類を誤分類 | `question_id`、`category`、`field_path`に基づく評価 |
| HTML rendererによる再推論 | 文書出力段階で被害類型・証拠状態が変わる可能性 | `DocumentViewModel`へ統一 |
| 文書必須項目の不足 | 被害救済／届出書類に必要なフィールドが欠落 | Document Completion Engineと不足フィールド計算 |
| 内部処理メッセージの露出 | 「local分析完了」などの内部文言がユーザーへ表示 | ユーザー向け文言と内部ログを分離 |

評価の中心は、モデルが魅力的な回答を生成したかではなく、銀行担当者が確認可能な受付パッケージを作成できたかです。

## 7. Key Design Decisions

### LLMを判断者ではなく事件整理ツールに限定した

ボイスフィッシング被害は、支払停止、警察届出、通信会社への届出、名義盗用確認が同時に必要になり得る緊急事件です。  
したがって、Recover24は112への届出や銀行の正式手続を代替しません。

LLMは被害者の自然言語による申告から事件情報を抽出するためにのみ使用しました。  
被害救済の可否、銀行責任、支払停止の可否などの機微な判断はLLMに直接断定させず、構造化case、ルールベース確認、人による確認へ分離しました。

### 事件全体を再生成せず、必要なフィールドだけを修正した

追加回答が入るたびに事件全体を再生成すると、既に入力された被害金額、取引内容、証拠状態が失われる可能性があります。

そこで`RecoveryCaseV1`を一つの事件記録として維持し、追加回答は当該質問が指すフィールドのみを修正するよう設計しました。  
例えば被害認知日への回答が入った場合、日付フィールドだけを修正し、既存の取引内容と証拠状態は維持します。

### 質問文ではなく質問IDで追加回答を処理した

ユーザーに表示する質問文はいつでも変更される可能性があります。  
文章中の単語から質問種類を推測すると、文言が少し変わっただけで回答処理や評価が不安定になる可能性があります。

そこで各質問に`question_id`、`handler_id`、`field_path`を付与しました。  
これにより質問文が変わっても、日付質問、取引内容質問、証拠質問などを安定して処理できます。

### 文書出力前に表示用データを固定した

警察届出用文書、銀行相談メモ、被害救済申請補助資料は、同じ事件情報を基に作成される必要があります。

各文書生成器が事件を直接再解釈すると、文書ごとに被害類型や証拠状態が異なって表示される可能性があります。  
これを防ぐため、文書出力前に`DocumentViewModel`を生成し、すべての文書とReviewer Consoleが同じ表示用データを使用するようにしました。

### 評価は質問文ではなく構造化結果を基準にした

質問文はユーザー体験改善のため変更される可能性があります。  
そのため、評価器が質問文そのものを見て判断すると、実際の機能が同じでも文言変更だけで評価が失敗する可能性があります。

そこで評価は`question_id`、`category`、`field_path`、不足フィールド、文書完成度など、構造化された出力値を基準に行いました。  
これによりユーザー向け文言を改善しても、中心機能が維持されているかを安定して確認できます。

## 8. Development Notes

本プロジェクトは当初、「被害者の申告を受け取り、書類を作成するチャットボット」のように見えました。

しかし開発を進める中で、中心的な問題はチャットボットの回答生成ではなく、**機微な金融被害事件を安全に構造化し、不足なく受付可能な状態へ整えること**だと明確になりました。

第一の転換点は質問設計でした。初期には振込型被害を中心に質問が固定される問題がありました。しかし実際のボイスフィッシングには、不正アプリ、名義盗用、カードローン、携帯電話開通、口座開設、現金引き渡し、融資詐欺など複数の流れが混在します。そこで事件類型と不足フィールドに応じて次の質問を動的に生成する方向へ変更しました。

第二の転換点は追加回答の反映方法でした。case全体を再生成する方法は実装が容易でしたが、既に入力された証拠状態や同意値が損傷するリスクがありました。そこで質問の`handler_id`に基づき、特定フィールドのみをpatchする構造へ変更しました。

第三の転換点は文書出力でした。単に見栄えのよいHTMLを作るのではなく、すべての文書が同じ事件状態と同じ表示モデルを使用する必要がありました。そこで`DocumentViewModel`を一元化しました。

第四の転換点は評価方式でした。LLMの回答が自然に見えることより、中心事実が欠落せず、禁止質問を行わず、文書提出準備度が改善されることの方が重要でした。そのため、評価基準をチャットボット品質から受付業務の完成度へ変更しました。

最終的にRecover24は、「被害者相談チャットボット」から、**銀行バックオフィスの確認まで接続する被害受付ワークフローエンジン**として整理されました。

## 9. Limitations

Recover24はポートフォリオ向けプロトタイプであり、実際の銀行運用システムとして使用するには複数の制約があります。

第一に、銀行・警察・被害救済の正式様式への座標マッピングは完全ではありません。現在の文書は提出補助用の例示様式であり、実際の提出前には各機関の最新様式と案内を確認する必要があります。

第二に、顧客本人確認と実際の認証連携は実装していません。実サービスでは、本人確認、口座認証、携帯電話認証、代理申請権限の確認が必要です。

第三に、tenant／権限／証拠アップロードはモデルまたはメタデータ段階です。実運用ではDB、オブジェクトストレージ、マルウェア検査、アクセス権限、保存期間ポリシーが必要です。

第四に、個人情報と機微な金融情報を扱うため、保存時暗号化、マスキング、アクセスログ、保存期間、削除ポリシーが必須です。

第五に、実際の銀行内部システム、警察届出システム、通信会社の名義盗用届出システムとは連携していません。

第六に、LLM抽出結果は常に検証が必要です。Recover24は自動判断システムではなく、担当者確認を前提とした受付支援システムです。

第七に、評価ゴールデンセットは実運用の分布全体を代表するものではありません。より多様な被害類型、最新手口、地域別・年齢別事例、実際の相談ログに基づく検証が必要です。

## 10. How to Run

### Install dependencies

```bash
pip install -r requirements.txt
```

### Set environment variables

Geminiを使用する場合：

```powershell
$env:GEMINI_API_KEY="your-api-key"
$env:RECOVER24_LLM_MODEL="gemini-3.1-flash-lite-preview"
```

Gemma Colab providerを使用する場合：

```powershell
$env:RECOVER24_LLM_PROVIDER="gemma_colab"
$env:RECOVER24_GEMMA_COLAB_URL="https://your-tunnel-url.trycloudflare.com"
```

ローカルテストではmock providerを使用し、API呼び出しなしで動作させることができます。

### Run tests

Windowsの一時フォルダ権限問題が発生する可能性があるため、`--basetemp`の使用を推奨します。

```powershell
python -m pytest tests -q -p no:cacheprovider --basetemp .pytest_tmp_local
```

### Run API

```powershell
uvicorn recover24.api_v1:create_app --factory --reload
```

最小API endpointは次のとおりです。

```text
GET  /health
POST /v1/cases/analyze
POST /v1/submissions
```

### Run a sample case

```powershell
PYTHONPATH=$PWD:$PYTHONPATH python application/run_recover24_case.py --input examples/sample_statement.txt --output-dir outputs/sample_case
```

### Run evaluation

```powershell
python scripts/run_conversation_gold_eval.py --engine llm --max-cases 5 --out outputs/eval_sample
```

## 12. What This Project Demonstrates

本プロジェクトは、LLMを機微な金融被害受付業務へ適用する際に必要なエンジニアリング設計を示します。

第一に、被害者の自然言語による申告を、構造化された事件JSONへ変換するinformation extractionフローを設計しました。

第二に、LLM抽出と、ルールベースの不足フィールド計算、文書準備度判断、人による確認を分離しました。

第三に、`RecoveryCaseV1`を単一の事件状態とし、追加回答は対象フィールドのみをpatchするよう設計しました。

第四に、質問へ`question_id`、`handler_id`、`category`、`field_path`を付与し、質問文と処理ロジックを分離しました。

第五に、文書出力は`DocumentViewModel`を通じて統一し、rendererが事件を再推論しないようにしました。

第六に、被害救済・警察届出・銀行相談に必要な情報を、文書パッケージとreviewer console viewへ整理しました。

第七に、評価基準を一般的なチャットボット品質ではなく、必須事実抽出、不足質問、文書完成度、禁止質問回避、監査可能性として定義しました。

最後に、自動判断ではなく、人による確認を前提とする安全な金融被害受付ワークフローとして設計しました。

本プロジェクトの中心は、単にLLMチャットボットを作成したことではなく、**ボイスフィッシング被害者の非構造化された申告を、銀行担当者が確認可能な事件パッケージへ変換する、安全なLLM workflowを設計したこと**です。
